## 🤖 UI-CHAT: AI як Третій Адаптер (OLMUI Triad)

> **Принцип**: `ui-chat` — не окремий продукт, а третій UI-шар для OLMUI.
> `ui-cli` (термінал) → `ui-lit` або `ui-react` (веб) → **`ui-chat` (AI)**

### Архітектура

Кожен додаток вже описує свої можливості через `ui-cli`:

- **Меню** (`adapter.select()`) = доступні дії
- **Вхідні дані** (`adapter.ask()`) = форми / параметри
- **Бізнес-логіка** (`domain/Model`) = валідація та обробка
- **Результат** (`adapter.log()`) = вивід

AI Chat **не дублює** цю логіку. Замість кастомних `tools` чи `function calling`,
LLM отримує descriptor меню з `ui-cli` і викликає існуючі дії через адаптер:

```js
// Термінал (людина натискає кнопки):
run(cliAdapter) // adapter.select → prompts, adapter.ask → readline

// AI Chat (LLM "натискає кнопки"):
run(aiAdapter) // adapter.select → LLM обирає опцію, adapter.ask → LLM відповідає
```

### Правила

1. **Не дублювати**: Забороняється створювати окремі `TOOLS` для AI, якщо та ж функція вже є в CLI.
2. **Адаптер = Інтерфейс**: `ask()`, `select()`, `log()`, `write()` — єдиний контракт для всіх UI.
3. **Валідація в Domain**: Бізнес-правила (вік > 18, сума < ліміт) описані в моделях домену, не в промптах.
4. **Locale-Aware**: AI отримує `locale` і відповідає мовою інтерфейсу.
5. **Zero Extra Code**: Щоб додати AI чат у новий додаток — достатньо `run(aiAdapter)`.

## 🧱 ІНТЕРФЕЙС МЕТА-ОПИСУ (META-DESCRIPTION INTERFACE)

Компоненти мають описуватися з використанням високорівневих семантичних назв:

- **Формат**: `App.ModuleName.ComponentName`
- **Пропси**: Використовуйте природні назви (наприклад, `$branch`, `$view`).
- **Приклад**:

  ```yaml
  - App.Branches.Card: true
    $branch: $data.current

  # або
  - App.Branches.Map:
      view: 'map'
  ```

## 🏗️ DATA-DRIVEN UI (nan0web Architecture)

Це ключовий архітектурний паттерн екосистеми.

1.  **Єдине Джерело Правди**:
    - Вся бізнес-логіка та контент визначаються у **YAML/JSON файлах** (`data/`).
    - UI-компоненти лише відображають ці дані ("Dumb Components").
2.  **BrowserDB & Inheritance**:
    - Використовуйте `BrowserDB` для роботи з даними.
    - **Inheritance**: `db.fetch('uk/metaly/_')` автоматично мерджить дані батьківських `_` файлів.
3.  **$content Pattern**:
    - Сторінка описується полем `$content` у YAML.
    - `UIReact` автоматично парсить `$content`.
    - Приклад:
      ```yaml
      $content:
        - App.Metals.Catalog: true
      ```
4.  **Entry Point**:
    - Використовуйте `UIRoot`.
5.  **Dynamic Hub Routing (In-Memory File Routing)**:
    - **Шлях = Роут**: Сам шлях до YAML/JSON файлу в структурі автоматично є його URL-адресою (роутом).
    - **Ефективність для AI (Agent-First)**: Багатомовність та створення нових сторінок полягає у фізичному створенні "дзеркальних" файлів. Це неймовірно проста і надійна задача для LLM з правильним контекстом.
    - **In-Memory Масштабування**: Навіть екосистема зі 100 000 файлів займає ~1 ГБ — легко утримується повністю in-memory для космічної швидкості доступу.
6.  **Плоска Екстракція i18n (`Model.UI`)**:
    - Будь-які текстові значення, вкладені всередині статичного об'єкту моделі, назва якого починається на `ui` (нечутливо до регістру, наприклад `UI` або `UI_TEXT`), автоматично скануються (`@nan0web/i18n`) та повністю "флетяться" на ключі для словника перекладу.

## 🎨 ZERO-LOGIC STYLE INJECTION (Style Entrypoint)

> **ОБОВ'ЯЗКОВО** для всіх додатків з веб-інтерфейсом (`ui-react`, `ui-lit`).

### Принцип

Кожен додаток **виділяє** всі CSS/SCSS-імпорти в окремий файл-точку входу `theme.js` (або `styles.js`):

```js
// src/ui/react-bootstrap/theme.js — ЄДИНЕ ДЖЕРЕЛО ПРАВДИ ДЛЯ СТИЛІВ
import 'bootstrap/dist/css/bootstrap.min.css'
import './styles.css'
```

Головний `main.jsx` підключає стилі через нього:

```js
import './theme.js'   // ← стилі
import App from './App.jsx'  // ← логіка
```

### Навіщо

Розділення CSS на окрему точку входу дозволяє **будь-якому зовнішньому інструменту** (SSG Gallery, Sandbox, Storybook, E2E-тест) підключити **100% стилів додатка** без запуску React, Router чи Store:

```html
<!-- Iframe / Sandbox / SSG Viewer -->
<script type="module" src="/src/ui/react-bootstrap/theme.js"></script>
```

Vite компілює CSS-залежності з `theme.js` і нативно інжектує стилі у `<head>`, забезпечуючи:
- ✅ **100% Fidelity** — стилі ідентичні реальному додатку
- ✅ **Vite HMR** — зміна в CSS миттєво оновлює preview
- ✅ **Zero Logic Overhead** — жодного React, API чи бізнес-логіки

### Правила

1. **Один файл — одне призначення:** `theme.js` містить **тільки** `import` стилів. Жодної логіки.
2. **main.jsx споживає theme.js:** Головна точка входу імпортує `theme.js`, а не окремі CSS-файли напряму.
3. **Хардкод CDN заборонений:** Інструменти візуалізації (SSG Gallery, Sandbox) підключають стилі **виключно** через `theme.js`.
4. **Framework Agnostic:** Патерн працює з Bootstrap, Tailwind, Lit CSS, будь-чим — головне, що імпорти зібрані в одному місці.

## 📋 РЕГЛАМЕНТ ТЕСТУВАННЯ ТА ВИПРАВЛЕННЯ БАГІВ (User Story Testing Rule)

> **КОНТРАКТ НЕПРЕРЕРВНОГО ВДОСКОНАЛЕННЯ**: Жоден баг не повинен бути виправленим без сценарного тестування.

1. **User Story Definition**: Кожне зауваження користувача, або знайдений баг під час ручного чи автоматизованого тестування, **ОБОВ'ЯЗКОВО** повинне бути формалізоване як опис (User Story або User Remark).
2. **Synchronized Coverage (Red-Green-Refactor)**: 
   - Відразу після знаходження багу, **до его виправлення**, створюється тест (на рівні моделі `xxxModel.test.js`), який чітко відтворює і показує цю помилку (світло-червоний / failed).
   - Лише після перевірки падіння тесту виконується фікс логіки, щоб тест вийшов на "зелене світло" (passed).
   - Для глобального регресійного контролю обов'язково використовуються скрипти `/check`, `/check-all` та команди `npm run test:all`.
3. **In-Memory DB Simulation**: Використовуйте `new DB({ predefined: [] })` для імітації пам'яті та файлової системи (`db.listDir`, віртуальні маунти), щоб відтворити поведінку багу і провести верифікацію без запуску реального процесу чи клієнта. Так ми гарантовано позбуваємося повторного виникнення одних і тих самих помилок.
