# Системні інструкції `system.md` / NaN•Web UI

Читайте також [загальні інструкції nan0web](../../system.md)

Чистий JavaScript для створення інтерфейсів користувача без залежностей від платформи.
Повинен працювати в Node.js, браузері та інших середовищах, що підтримують ESM.

- статичні базові функції (static base functions)
- статичні асинхронні функції (static async functions)

  1.5. Представлення (Views) та Компоненти:
  1.5.1. Повинні бути ізольованими функціями або класами, що приймають об'єкт даних (Data Object).

### 2. Тести та Верифікація

Кожна модель, компонент або клас повинні мати власний файл тестів для покриття функціональності в ізольованому середовищі.
З версії v1.11.0 основним стандартом тестування логіки додатків є **ScenarioTest** (детерміноване тестування інтентів).

#### node:test

Для ядра (base), cgi та cli інтерфейсів, а також веб-компонентів без `.jsx` слід використовувати `node:test` з `node:assert/strict`.

## Архітектура OLMUI

1. `base` - Базові доменні моделі та інтенти.
1. `cli` - Консольний додаток на основі `@nan0web/ui-cli`.
1. `web` - Веб-інтерфейс на основі `Lit` або `React`.
1. `mobile` - Мобільний інтерфейс (Swift/Kotlin).

### Model-as-App та Intents (v1.12.0+)

Логіка додатків реалізується через генератори, що повертають **Наміри (Intents)**. Це дозволяє логіці бути повністю незалежною від способу вводу/виводу.

#### Ввід (ask) та Вивід (show)

Замість прямої взаємодії з потоками (stdin/stdout), додаток використовує генератори інтентів:

```js
import { ModelAsApp, ask, show, result } from '@nan0web/ui'

class UserOnboardingApp extends ModelAsApp {
  async *run() {
    // 1. Запит даних (ask) - адаптер сам вирішить, як це показати (Input у CLI чи Modal у Web)
    const name = yield ask('name', { help: 'Ваше ім\'я', required: true })
    
    if (name.value === 'admin') {
      yield show('Доступ обмежено', 'error')
      return result({ ok: false })
    }

    // 2. Рендеринг компонента
    yield show(`Привіт, ${name.value}!`, 'info')

    return result({ ok: true, user: name.value })
  }
}
```

### Представлення (Views) та Дані

#### Model-as-Schema

Ми використовуємо моделі як схеми даних. Компоненти приймають екземпляри моделей, що гарантує нормалізацію та типізацію даних.

```js
import { HeaderModel } from '@nan0web/ui/models'

// Функція-компонент (Pure View)
function HeaderView(model) {
  if (!(model instanceof HeaderModel)) {
    throw new TypeError('Очікується HeaderModel')
  }
  
  // Рендеринг на основі властивостей моделі
  return Frame.from([[model.title, model.logo]])
}
```

## Крос-платформність (Capability Discovery)

В архітектурі **NaN•Web OLMUI** додаток — це набір моделей.

1. **Додаток як Джерело Правди**: Додаток сам експортує свої можливості через `static metadata`.
2. **Data-Driven Decoupling**: UI нічого не знає про логіку. Він лише відображає стан моделі та відправляє наміри назад.

### Формула y = f(x)

Робота системи зводиться до:
- **`f` (Логіка)** — генератор `ModelAsApp.run()`.
- **`x` (Стан)** — вхідні дані моделі.
- **`y` (Інтенти)** — потік `Intents` (ask, show, render), що інтерпретується UI-адаптером.

Це забезпечує наявність **однієї логіки (f)** для безлічі **інтерфейсів (UI)**.
